# Техническое задание для фронтенда: внешние агенты и заявки на объявления

## 1. Цель

Реализовать фронтенд-интерфейс для работы с внешними агентами, которые подают легкие заявки на объявления, и для внутренних агентов, которые принимают эти заявки, проверяют их, дополняют и быстро создают полноценные объявления в существующей CRM.

Фронтенд должен опираться на уже реализованный backend-функционал внешних заявок, не дублировать бизнес-логику статусов на клиенте и использовать существующие сущности проекта: пользователей, роли, объекты недвижимости, типы недвижимости, локации, ремонты, филиалы, группы филиалов и внутренние агентские роли.

## 2. Термины

- **Внешний агент** - пользователь с ролью `external_agent`. Он не работает в CRM как внутренний агент, а только подает заявки на объекты и смотрит их дальнейший статус.
- **Заявка внешнего агента** - легкая форма объявления, которую заполняет внешний агент. На backend это сущность `external_property_requests`.
- **Внутренний агент** - сотрудник CRM, который может взять заявку в работу, проверить данные, дополнить их и создать полноценное объявление.
- **Конвертация** - создание обычного объекта недвижимости из заявки внешнего агента.
- **Источник объявления** - связь созданного объекта с заявкой и внешним агентом, чтобы в будущем видеть, кто подал первичную информацию.

## 3. Роли и доступы

### 3.1. Внешний агент

Пользователь с ролью `external_agent` должен видеть только свой кабинет внешнего агента:

- список своих заявок;
- создание черновика заявки;
- редактирование своей заявки до момента активной внутренней обработки;
- отправку заявки на проверку;
- загрузку и удаление фото;
- просмотр статуса заявки;
- просмотр публичного статуса созданного объявления, если заявка была конвертирована.

Внешний агент не должен видеть:

- внутреннюю CRM;
- список всех объектов;
- внутренние контакты и комментарии;
- отчеты, аналитику, сотрудников, филиалы и системные разделы;
- заявки других внешних агентов.

### 3.2. Внутренние пользователи CRM

Внутренние пользователи должны видеть раздел "Заявки внешних агентов" согласно backend-доступам. Фронтенд должен скрывать недоступные разделы в навигации, но финальную проверку прав считать ответственностью backend.

Внутренний пользователь может:

- видеть очередь заявок;
- фильтровать заявки;
- открывать детальную карточку заявки;
- назначать ответственного агента;
- менять рабочий статус заявки;
- видеть данные внешнего агента;
- видеть контакты владельца, если backend вернул эти данные;
- получить предзаполнение для формы объекта;
- создать полноценное объявление из заявки;
- видеть метку источника у созданного объекта.

## 4. Навигация

### 4.1. Для внешнего агента

После входа внешний агент должен попадать на страницу:

`/external/property-requests`

Название в интерфейсе: **Мои заявки**.

Навигация внешнего агента минимальная:

- Мои заявки;
- Новая заявка;
- Профиль;
- Выход.

Не нужно показывать маркетинговую посадочную страницу или общий dashboard CRM. Первый экран должен быть рабочим интерфейсом со списком заявок.

### 4.2. Для внутренних пользователей

В CRM добавить раздел:

`/crm/external-agent-requests`

Варианты размещения:

- в меню "Объекты";
- или отдельным пунктом "Внешние заявки";
- или в группе "CRM / Заявки".

Рекомендуемое название: **Заявки внешних агентов**.

В карточке и списке обычных объектов недвижимости добавить визуальную метку источника, если объект создан из заявки внешнего агента.

## 5. Backend API

Фронтенд должен использовать существующие endpoint-ы.

### 5.1. Кабинет внешнего агента

Все endpoint-ы требуют авторизацию Sanctum и активного пользователя.

```http
GET    /api/external/property-requests
POST   /api/external/property-requests
GET    /api/external/property-requests/stats
GET    /api/external/property-requests/{id}
PATCH  /api/external/property-requests/{id}
POST   /api/external/property-requests/{id}/submit
POST   /api/external/property-requests/{id}/photos
DELETE /api/external/property-requests/{id}/photos/{photoId}
```

Создание черновика:

```http
POST /api/external/property-requests?draft=1
```

### 5.2. Внутренний интерфейс CRM

```http
GET   /api/external-agent-requests
GET   /api/external-agent-requests/stats
GET   /api/external-agent-requests/leaderboard
GET   /api/external-agent-requests/{id}
PATCH /api/external-agent-requests/{id}/assign
PATCH /api/external-agent-requests/{id}/status
GET   /api/external-agent-requests/{id}/prefill
POST  /api/external-agent-requests/{id}/convert
```

### 5.3. Интеграция с объектами

```http
GET /api/properties?source_type=external_agent
GET /api/properties?external_agent_id={userId}
GET /api/properties/{id}
```

В авторизованной карточке объекта backend возвращает `external_source`, если объект создан из заявки внешнего агента. В публичной карточке объекта этот блок не должен отображаться.

## 6. Статусы заявок

Фронтенд должен отображать backend-статусы без самостоятельного пересчета бизнес-логики.

| Status | Название | Цвет |
| --- | --- | --- |
| `draft` | Черновик | Серый |
| `submitted` | Отправлена | Синий |
| `assigned` | Назначена | Индиго |
| `in_review` | В проверке | Желтый |
| `needs_info` | Нужна информация | Оранжевый |
| `duplicate` | Дубликат | Красный/желтый |
| `rejected` | Отклонена | Красный |
| `converted` | Создано объявление | Зеленый |
| `archived` | Архив | Серый |

Для внешнего агента backend может вернуть упрощенный `display_status`. Его надо показывать как основной публичный статус, если поле присутствует.

| Display status | Название для внешнего агента |
| --- | --- |
| `draft` | Черновик |
| `submitted` | Отправлена |
| `in_work` | В работе |
| `needs_info` | Нужна информация |
| `property_created` | Объявление создано |
| `published` | Объявление опубликовано |
| `property_rejected` | Отклонено |
| `closed_deal` | Сделка закрыта |

## 7. Кабинет внешнего агента

### 7.1. Страница "Мои заявки"

URL:

`/external/property-requests`

Данные:

```http
GET /api/external/property-requests
GET /api/external/property-requests/stats
```

На странице показать:

- карточки статистики: всего, черновики, отправленные, в работе, созданные объявления, отклоненные;
- таблицу или список заявок;
- кнопку "Новая заявка";
- фильтр по статусу;
- поиск по адресу, району, телефону владельца или тексту, если backend поддерживает параметр `q`;
- пагинацию;
- пустое состояние, если заявок еще нет.

Колонки списка:

- номер заявки;
- тип сделки;
- тип недвижимости;
- локация/район;
- цена;
- статус;
- дата создания;
- дата последнего изменения;
- действие "Открыть".

На мобильном экране список должен превращаться в компактные карточки.

### 7.2. Создание заявки

URL:

`/external/property-requests/new`

Форма должна быть легкой, но структурированной. Ее задача - помочь внешнему агенту быстро подать информацию, даже если данные неполные.

Поля:

- `offer_type` - тип сделки;
- `type_id` - тип недвижимости;
- `location_id` - город/локация;
- `district` - район;
- `address` - адрес;
- `landmark` - ориентир;
- `price` - цена;
- `currency` - валюта;
- `rooms` - количество комнат;
- `total_area` - общая площадь;
- `living_area` - жилая площадь;
- `land_size` - площадь земли;
- `floor` - этаж;
- `total_floors` - этажность дома;
- `repair_type_id` - тип ремонта;
- `condition` - состояние;
- `owner_name` - имя владельца;
- `owner_phone` - телефон владельца;
- `external_comment` - комментарий внешнего агента.

Справочники брать из существующих API проекта, например:

```http
GET /api/property-types
GET /api/locations
GET /api/repair-types
```

Если в текущем фронте эти справочники уже загружаются для формы обычного объекта, нужно переиспользовать существующие hooks/services/components.

### 7.3. Черновик и отправка

Форма должна поддерживать два сценария:

1. **Сохранить черновик**
   - отправить `POST /api/external/property-requests?draft=1`;
   - разрешить неполные данные;
   - показать статус "Черновик";
   - оставить возможность редактирования.

2. **Отправить на проверку**
   - отправить `POST /api/external/property-requests`;
   - или для уже созданного черновика `POST /api/external/property-requests/{id}/submit`;
   - после успешной отправки перевести пользователя на детальную страницу заявки.

Минимально обязательные поля для отправки не-черновика:

- `offer_type`;
- `type_id`;
- `price`;
- `currency`;
- `owner_phone`.

Фронтенд должен валидировать эти поля до отправки, но источник истины - backend. Ошибки backend надо показывать около соответствующих полей.

Формат ошибки может быть:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Validation failed.",
  "details": {
    "errors": {
      "offer_type": ["Поле обязательно."]
    }
  },
  "trace_id": "..."
}
```

Также нужно поддержать стандартный формат Laravel validation errors, если он используется в других endpoint-ах проекта.

### 7.4. Редактирование заявки

URL:

`/external/property-requests/{id}/edit`

Endpoint:

```http
PATCH /api/external/property-requests/{id}
```

Редактирование доступно, если backend разрешает изменение заявки. Если backend вернул `403`, показать понятное сообщение:

"Заявка уже в работе. Для изменения данных дождитесь запроса от внутреннего агента или создайте новую заявку."

Фронтенд не должен самостоятельно решать, какие статусы редактируемые. Можно визуально отключать кнопку редактирования для очевидно финальных статусов, но итоговое решение остается за backend.

### 7.5. Детальная страница заявки

URL:

`/external/property-requests/{id}`

Endpoint:

```http
GET /api/external/property-requests/{id}
```

Показать:

- статус или `display_status`;
- основную информацию об объекте;
- контакты владельца, которые сам внешний агент указал;
- комментарий внешнего агента;
- фото;
- дату создания и обновления;
- безопасную историю статусов, если backend ее вернул;
- причину отклонения или запрос дополнительной информации, если применимо;
- блок созданного объявления, если заявка конвертирована.

Если есть связанный объект недвижимости:

- показать номер объекта;
- показать публичную ссылку, если backend возвращает ее или если фронт умеет ее собрать из данных объекта;
- показать статус публикации в понятном виде.

Не показывать внешнему агенту:

- внутренний комментарий;
- ответственного внутреннего агента, если это не предусмотрено backend;
- служебные логи;
- данные других агентов.

### 7.6. Фото

Endpoint-ы:

```http
POST   /api/external/property-requests/{id}/photos
DELETE /api/external/property-requests/{id}/photos/{photoId}
```

Требования:

- загрузка через `multipart/form-data`;
- поле `photos[]`;
- поддержать множественную загрузку;
- показать progress upload;
- показать preview до/после загрузки;
- обработать ошибки размера/формата;
- разрешить удаление только если backend разрешает;
- после загрузки обновить карточку заявки.

Рекомендуемые ограничения интерфейса:

- подсказка по форматам: JPG, PNG, WebP;
- подсказка по максимальному размеру, если backend возвращает лимит или он известен из проекта;
- drag-and-drop на desktop;
- обычный picker на mobile.

## 8. Внутренний раздел CRM

### 8.1. Очередь заявок

URL:

`/crm/external-agent-requests`

Endpoint-ы:

```http
GET /api/external-agent-requests
GET /api/external-agent-requests/stats
GET /api/external-agent-requests/leaderboard
```

На странице показать:

- статистику по статусам;
- таблицу заявок;
- фильтры;
- поиск;
- блок leaderboard внешних агентов;
- быстрые действия для назначения и открытия заявки.

Фильтры:

- `status`;
- `external_agent_id`;
- `assigned_agent_id`;
- `branch_id`;
- `branch_group_id`;
- `date_from`;
- `date_to`;
- `has_duplicate`;
- `q`.

Колонки:

- номер заявки;
- статус;
- внешний агент;
- тип сделки;
- тип недвижимости;
- локация/район;
- цена;
- владелец/телефон;
- назначенный внутренний агент;
- дата подачи;
- дата последнего изменения;
- действие "Открыть".

В таблице должны быть устойчивые состояния:

- загрузка;
- пустой результат;
- ошибка доступа;
- ошибка сети;
- повторить запрос.

### 8.2. Детальная карточка внутренней заявки

URL:

`/crm/external-agent-requests/{id}`

Endpoint:

```http
GET /api/external-agent-requests/{id}
```

Структура страницы:

- верхняя панель со статусом и основными действиями;
- блок данных внешнего агента;
- блок данных объекта;
- блок контактов владельца;
- фото;
- комментарий внешнего агента;
- внутренний комментарий, если backend возвращает это поле;
- история статусов;
- возможный дубль;
- связанный созданный объект, если заявка уже конвертирована.

Главные действия:

- назначить ответственного;
- изменить статус;
- запросить информацию;
- отметить как дубликат;
- отклонить;
- открыть предзаполнение;
- создать объявление.

Если backend не предоставляет отдельный endpoint для сохранения `internal_comment`, фронтенд не должен делать фиктивное сохранение. Поле можно показывать только как read-only или сохранять вместе со сменой статуса, если это поддержано контрактом конкретного endpoint-а.

### 8.3. Назначение ответственного

Endpoint:

```http
PATCH /api/external-agent-requests/{id}/assign
```

UI:

- selector внутреннего агента;
- подтверждение назначения;
- optimistic update допустим только с rollback при ошибке;
- после успешного назначения обновить заявку и список.

Если backend вернул `403`, показать:

"У вас нет прав назначать ответственного по этой заявке."

### 8.4. Смена статуса

Endpoint:

```http
PATCH /api/external-agent-requests/{id}/status
```

UI должен поддерживать смену статусов:

- `submitted`;
- `assigned`;
- `in_review`;
- `needs_info`;
- `duplicate`;
- `rejected`;
- `archived`.

Для отдельных статусов обязательны дополнительные поля на уровне интерфейса:

- `needs_info` - комментарий, какую информацию нужно уточнить;
- `rejected` - причина отклонения;
- `duplicate` - ссылка/выбор существующего объекта, если backend поддерживает `duplicate_property_id`;
- `archived` - подтверждение действия.

После смены статуса:

- обновить карточку заявки;
- обновить счетчики;
- показать toast об успехе;
- если ошибка `422`, подсветить поля модального окна.

### 8.5. Предзаполнение объявления

Endpoint:

```http
GET /api/external-agent-requests/{id}/prefill
```

Сценарий:

1. Внутренний агент нажимает "Создать объявление".
2. Фронтенд запрашивает `prefill`.
3. Открывается существующая форма создания объекта недвижимости.
4. Поля формы заполняются данными из заявки.
5. Внутренний агент дополняет недостающие данные.
6. Фронтенд отправляет создание через endpoint конвертации заявки, а не через обычное создание объекта.

Важно: форма создания объекта должна максимально переиспользовать существующий UI обычного создания недвижимости. Не нужно делать вторую независимую форму, если в проекте уже есть полноценная форма объекта.

### 8.6. Конвертация заявки в объект

Endpoint:

```http
POST /api/external-agent-requests/{id}/convert
```

Payload должен содержать отредактированные поля объекта недвижимости и дополнительные параметры:

```json
{
  "property": {
    "offer_type": "sale",
    "type_id": 1,
    "location_id": 2,
    "price": 120000,
    "currency": "USD"
  },
  "copy_photos": true,
  "force": false
}
```

Точное имя контейнера полей объекта должно соответствовать backend-контракту. Если текущий backend принимает поля объекта на верхнем уровне, фронтенд должен использовать фактический контракт из API-документации `docs/external-property-requests-api.md`.

После успешной конвертации:

- показать toast "Объявление создано";
- перевести заявку в статус `converted`;
- открыть карточку созданного объекта или показать ссылку на нее;
- обновить список заявок;
- в объекте должна отображаться метка источника "Внешний агент".

### 8.7. Дубликаты

Если backend вернул `409` при конвертации или заявка имеет статус `duplicate`, фронтенд должен показать предупреждение:

"Похоже, такой объект уже есть в базе."

В модальном окне показать:

- найденный похожий объект, если backend вернул его;
- адрес/район/цену/телефон для сравнения;
- кнопку "Открыть существующий объект";
- кнопку "Все равно создать" только для пользователей, которым backend это разрешит.

Для принудительного создания отправить:

```json
{
  "force": true
}
```

Фронтенд должен требовать явного подтверждения перед `force=true`.

## 9. Интеграция с карточкой объекта

### 9.1. Список объектов

В существующем списке объектов добавить фильтры:

- источник: все / обычные / внешний агент;
- внешний агент, если выбран источник "внешний агент".

Endpoint-ы:

```http
GET /api/properties?source_type=external_agent
GET /api/properties?external_agent_id={userId}
```

В карточке или строке объекта показать бейдж:

**Внешний агент**

Если backend вернул имя внешнего агента, tooltip или secondary text:

"Заявку подал: {name}"

### 9.2. Детальная карточка объекта

В авторизованной CRM-карточке объекта показать блок:

**Источник объявления**

Поля:

- тип источника: внешний агент;
- имя внешнего агента;
- телефон/email внешнего агента, если backend вернул;
- номер исходной заявки;
- ссылка "Открыть заявку";
- дата подачи заявки.

В публичной карточке объекта этот блок не показывать.

## 10. UI и UX требования

### 10.1. Общие принципы

- Использовать существующий дизайн-системный набор компонентов проекта.
- Не создавать отдельную визуальную стилистику для внешнего кабинета.
- Не делать landing page. Пользователь после входа должен попадать в рабочий интерфейс.
- Формы должны быть короткими визуально: группировать поля по смыслу и использовать раскрываемые секции для необязательных деталей.
- На mobile все ключевые сценарии должны быть доступны без горизонтального скролла.
- Действия с риском потери данных должны иметь подтверждение.

### 10.2. Рекомендуемая группировка формы внешней заявки

1. Тип объявления
   - тип сделки;
   - тип недвижимости.

2. Где находится объект
   - город/локация;
   - район;
   - адрес;
   - ориентир.

3. Цена и параметры
   - цена;
   - валюта;
   - комнаты;
   - площадь;
   - этаж.

4. Состояние
   - ремонт;
   - состояние.

5. Контакт владельца
   - имя;
   - телефон.

6. Комментарий и фото
   - комментарий;
   - загрузка фото.

### 10.3. Ошибки и состояния

Обязательные состояния:

- loading skeleton;
- empty state;
- validation error;
- network error;
- `401` - отправить на login;
- `403` - показать сообщение о недостатке прав;
- `404` - заявка не найдена;
- `409` - возможный дубликат;
- `422` - ошибки валидации;
- upload progress;
- upload failed with retry.

Ошибки backend с `trace_id` показывать пользователю кратко, а `trace_id` выводить в деталях ошибки или копируемом блоке для поддержки.

### 10.4. Toast и уведомления

Показывать короткие уведомления:

- черновик сохранен;
- заявка отправлена;
- фото загружены;
- фото удалено;
- ответственный назначен;
- статус изменен;
- объявление создано;
- действие не выполнено.

Не показывать длинные технические сообщения в toast. Подробности должны быть в форме или модальном окне.

## 11. API client и модели данных на фронте

### 11.1. Рекомендуемые сервисы

Создать или расширить API services:

- `externalPropertyRequestsApi` - кабинет внешнего агента;
- `externalAgentRequestsApi` - внутренний CRM-раздел;
- `propertiesApi` - фильтры и источник объекта;
- `dictionariesApi` - справочники формы.

Если в проекте уже есть единый API client, новые методы нужно добавить в него, не создавать параллельный HTTP-клиент.

### 11.2. Типы данных

На фронте нужны типы:

- `ExternalPropertyRequest`;
- `ExternalPropertyRequestPhoto`;
- `ExternalPropertyRequestLog`;
- `ExternalPropertyRequestStatus`;
- `ExternalPropertyRequestDisplayStatus`;
- `ExternalPropertyRequestStats`;
- `ExternalAgentLeaderboardItem`;
- `ExternalSource`;
- `ExternalPropertyRequestPrefill`.

В TypeScript проектах статусы должны быть union type, а не произвольная строка.

### 11.3. Пагинация

Списки заявок должны поддерживать Laravel pagination response. Фронтенд должен читать:

- `data`;
- `links`;
- `meta`;
- текущую страницу;
- последнюю страницу;
- общее количество.

Если текущий API client проекта уже нормализует пагинацию, использовать существующий формат.

## 12. Безопасность и приватность

- Не хранить лишние персональные данные владельцев в localStorage.
- Не кешировать детальные заявки с телефонами владельцев дольше текущей сессии.
- Не показывать внутренние комментарии внешнему агенту.
- Не показывать данные внешнего агента на публичных страницах объекта.
- Все действия должны идти через backend API, без клиентского обхода ролей.
- После logout очищать query/cache состояния, связанные с заявками и объектами.

## 15. Критерии приемки

Функционал считается готовым, если:

- внешний агент после входа видит только свой кабинет;
- внешний агент может создать черновик заявки;
- внешний агент может отправить заявку на проверку;
- внешний агент может загрузить и удалить фото;
- внешний агент видит понятный статус заявки;
- внешний агент не видит внутренние комментарии и CRM-данные;
- внутренний пользователь видит очередь заявок;
- внутренний пользователь может фильтровать заявки;
- внутренний пользователь может назначить ответственного;
- внутренний пользователь может менять статус заявки;
- внутренний пользователь может открыть предзаполнение объекта;
- внутренний пользователь может создать объект из заявки;
- при создании объекта сохраняется связь с внешним агентом;
- созданный объект имеет метку источника в CRM;
- публичная карточка объекта не раскрывает внешнего агента;
- ошибки `403`, `409`, `422` обработаны понятным интерфейсом;
- mobile-интерфейс не ломает основные сценарии;
- все новые экраны покрыты smoke/e2e тестами или проверочным чеклистом QA.

## 16. QA checklist

### Внешний агент

- войти под `external_agent`;
- убедиться, что CRM-разделы скрыты;
- создать пустой черновик;
- заполнить заявку минимальными данными;
- отправить заявку;
- проверить validation errors при пустых обязательных полях;
- загрузить несколько фото;
- удалить фото;
- открыть заявку после смены статуса внутренним агентом;
- проверить отображение статуса после конвертации.

### Внутренний агент

- открыть очередь заявок;
- проверить фильтр по статусу;
- проверить поиск;
- проверить фильтр по внешнему агенту;
- назначить ответственного;
- сменить статус на `needs_info`;
- сменить статус на `rejected` с причиной;
- открыть `prefill`;
- создать объект из заявки;
- проверить, что заявка стала `converted`;
- открыть созданный объект;
- проверить блок источника.

### Дубликаты

- создать заявку, похожую на существующий объект;
- попытаться конвертировать;
- получить предупреждение о дубле;
- открыть найденный объект;
- подтвердить принудительное создание, если роль имеет право;
- убедиться, что без подтверждения `force=true` не отправляется.

### Безопасность

- под внешним агентом попробовать открыть чужую заявку по прямому URL;
- под внешним агентом попробовать открыть `/crm/external-agent-requests`;
- под неавторизованным пользователем открыть API-зависимые страницы;
- проверить, что публичная карточка объекта не показывает `external_source`;
- проверить очистку состояния после logout.

## 17. Зависимости и открытые вопросы

Зависимости:

- существующий login/auth flow;
- существующие справочники типов недвижимости, локаций и ремонтов;
- существующая форма создания объекта;
- существующий UI kit таблиц, форм, модальных окон и upload-компонентов;
- backend API из `docs/external-property-requests-api.md`.

Открытые вопросы для фронтенд-реализации:

- точный URL публичной карточки объекта, если backend не возвращает готовую ссылку;
- финальный список ролей, которым разрешено принудительное создание дубля;
- нужен ли внешний агенту push/email/SMS notification при смене статуса;
- нужно ли сохранять внутренний комментарий отдельным endpoint-ом;
- какие лимиты фото использовать в UI, если они не возвращаются API.
