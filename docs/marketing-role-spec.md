# ТЗ: роль `Маркетолог`

## Цель
Добавить новую системную роль `marketing` (`Маркетолог`), которая по уровню доступа повторяет `admin`, но не имеет доступа ни к одному разделу и endpoint'у отчетов.

## Бизнес-правило
- Пользователь с ролью `marketing` должен иметь все права роли `admin`.
- Пользователь с ролью `marketing` не должен получать права `superadmin`.
- Пользователь с ролью `marketing` не должен иметь доступ к отчетам:
  - CRM-отчеты
  - отчеты по объектам
  - агентские отчеты
  - отчеты по показам
- Пользователь с ролью `marketing` не должен иметь права назначать пользователям роли `admin` и `superadmin`.
- Пользователи с ролями `rop` и `branch_director` не должны иметь права назначать роль `admin`.

## Backend scope
- Добавить роль `marketing` в таблицу `roles` через миграцию.
- Добавить роль `marketing` в сидер ролей.
- Включить `marketing` во все проверки privileged/admin-level доступа, где сейчас используется логика `admin`/`superadmin`, кроме report-flow.
- Явно запретить `marketing` во всех report-controller'ах и report-endpoint'ах.

## Backend acceptance criteria
- Роль `marketing` появляется в `/api/roles` и доступна для назначения пользователю.
- Пользователь с ролью `marketing` видит и управляет сущностями на уровне `admin`.
- Пользователь с ролью `marketing` может:
  - работать с пользователями
  - работать с группами филиалов
  - работать с CRM-сущностями и справочниками на уровне `admin`
- Пользователь с ролью `marketing` может назначать только роли:
  - `marketing`
  - `branch_director`
  - `rop`
  - `agent`
  - `manager`
  - `operator`
  - `client`
- Пользователь с ролью `marketing` не может открыть endpoints:
  - `/api/crm/reports/performance`
  - `/api/bookings/agents-report`
  - `/api/reports/*`
- При попытке открыть отчет backend возвращает `403 Forbidden`.

## Frontend scope
- Добавить отображение роли `Маркетолог` во всех формах/селектах выбора роли.
- В списках пользователей и карточке пользователя корректно показывать label роли `Маркетолог`.
- Скрыть пункты меню, вкладки, кнопки и прямые переходы в разделы отчетов для роли `marketing`.
- Если frontend получает `403` по report-endpoint, показывать штатное сообщение о недоступности раздела без падения страницы.

## Frontend acceptance criteria
- Роль `Маркетолог` доступна в UI при создании и редактировании пользователя.
- После логина под `marketing` пользователь видит тот же интерфейс, что и `admin`, кроме отчетов.
- В навигации отсутствуют ссылки на отчеты.
- Прямой переход по URL в отчет не ломает приложение: пользователь получает экран/нотификацию `Нет доступа`.
- В форме создания/редактирования пользователя для `marketing` недоступны роли `admin` и `superadmin`.
- В форме создания/редактирования пользователя для `rop` и `branch_director` недоступна роль `admin`.

## API contract
- Новый slug роли: `marketing`
- Название роли: `Маркетолог`
- Ожидаемое поведение для forbidden-отчетов: HTTP `403`

## Риски
- Если во frontend права завязаны на whitelist ролей, роль `marketing` может не попасть в существующие проверки и часть admin-функций останется скрыта.
- Если какие-то report-endpoint'ы не считаются “отчетами” только по URL, их нужно дополнительно проверить по фактическому использованию на frontend.

## Prompt для frontend
Реализуй поддержку новой роли пользователя `marketing` (`Маркетолог`) в административном интерфейсе. Роль `marketing` должна вести себя как `admin` во всех разделах приложения, кроме отчетов. Нужно:

1. Добавить роль `Маркетолог` во все формы создания/редактирования пользователя и во все UI-отображения роли.
2. Во всех клиентских проверках доступа считать `marketing` admin-level ролью, кроме правил назначения ролей и доступа к отчетам.
3. Для пользователя с ролью `marketing` скрыть из селекта назначения роли `admin` и `superadmin`.
4. Для пользователей с ролями `rop` и `branch_director` скрыть из селекта назначения роль `admin`.
5. Полностью скрыть для `marketing` разделы отчетов, пункты меню отчетов, кнопки перехода к отчетам и любые report widgets/shortcuts.
6. На прямом заходе по URL отчетов обрабатывать `403 Forbidden` и показывать корректное состояние `Нет доступа`, без white screen и без бесконечных retry.
7. Не менять поведение для `admin`, `superadmin` и остальных ролей.

Критерии приемки:
- `marketing` видит все admin-разделы, кроме отчетов.
- `marketing` не может назначить `admin` и `superadmin`.
- `rop` и `branch_director` не могут назначить `admin`.
- В меню и на dashboard нет ссылок на отчеты для `marketing`.
- Создание/редактирование пользователя с ролью `marketing` работает штатно.
- Любой запрос в report API под `marketing`, если он все же произошел, обрабатывается корректно через UI error state.
