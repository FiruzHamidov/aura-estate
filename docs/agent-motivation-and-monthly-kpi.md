# ТЗ: Система мотивации агентов, совместимая с текущей KPI/CRM моделью

## 1. Цель
Запустить мотивационную программу без ломки текущего KPI-модуля и текущих моделей данных:
- личные награды агента за продажи,
- командная награда всей компании,
- синхронизация с KPI и планами периода.

## 2. Фиксированные бизнес-правила (утверждено)
1. 5-я личная продажа агента -> поездка в Ташкент.
2. 8-я личная продажа агента -> поездка в Умру.
3. 100-я продажа всей компании Aura -> вечеринка на даче для всего Aura.

## 3. As-Is (что уже есть в системе)

### 3.1 Источник факта продаж
Факт продажи уже считается в CRM через:
- `properties.sold_at` (дата продажи),
- `properties.moderation_status in ('sold', 'rented', 'sold_by_owner')`.

Для атрибуции на агента уже используется:
- `property_agent_sales (property_id, agent_id, role, ...)`.
- Вклад агента при совместной продаже: `1 / N`, где `N` = число участников (до 3).
- Fallback на `properties.agent_id`, если для объекта нет записей в `property_agent_sales`.

Это реализовано в `DailyReportService::countDeals()`.

### 3.2 KPI, который уже работает
- Ежедневные KPI-данные: таблица `daily_reports`.
- Продажи в KPI:
  - `sales_count` (decimal, дробный вклад),
  - `deals_count` (legacy integer для обратной совместимости).
- Планы KPI: `kpi_plans` (поддерживаются user/branch/branch_group + effective_from/effective_to).
- API KPI уже доступны в `routes/api.php` (`/api/kpi/*`, `/api/kpi-reports`, `/api/kpi/daily/*`).

### 3.3 Важный вывод для ТЗ
Новая мотивация не должна пересчитывать продажи отдельно. Она должна использовать ту же бизнес-логику продажи, что и KPI (через `sold_at + moderation_status + property_agent_sales`).

## 4. To-Be архитектура (минимально инвазивно)

### 4.1 Переиспользуем существующие сущности
- `users` — агент.
- `properties` — факт продажи.
- `property_agent_sales` — участники продажи.
- `daily_reports` — база для KPI-агрегаций.
- `kpi_plans` — хранение планов периода.

### 4.2 Новые сущности (только для наград)

1. `motivation_rules`
- `id`
- `scope` enum: `agent|company`
- `metric_key` string (для текущего кейса: `sales_count`)
- `threshold_value` decimal(8,4) (5, 8, 100)
- `reward_type` enum: `trip_tashkent|trip_umra|company_party`
- `name`, `description`
- `is_active` boolean
- timestamps

2. `motivation_achievements`
- `id`
- `rule_id` fk -> `motivation_rules`
- `user_id` nullable fk -> `users` (для `agent`)
- `company_scope` boolean default `false` (для `company`)
- `threshold_reached_at` datetime
- `threshold_value_snapshot` decimal(8,4)
- `status` enum: `new|verified|approved|issued|cancelled`
- `meta` json
- timestamps
- уникальность:
  - `unique(rule_id, user_id)` для личных правил
  - `unique(rule_id, company_scope)` для company-правила

3. `motivation_reward_issues`
- `id`
- `achievement_id` fk
- `assignee_id` nullable fk -> `users` (HR/оператор)
- `status` enum: `new|in_progress|issued|rejected`
- `issued_at` nullable
- `comment` text nullable
- timestamps

## 5. Сидер стартовых мотиваций (обязательно)
Добавить сидер (например `MotivationRulesSeeder`) с 3 активными правилами:
1. `scope=agent`, `threshold_value=5`, `reward_type=trip_tashkent`, `name=5 продаж — Ташкент`.
2. `scope=agent`, `threshold_value=8`, `reward_type=trip_umra`, `name=8 продаж — Умра`.
3. `scope=company`, `threshold_value=100`, `reward_type=company_party`, `name=100 продаж компании — Вечеринка Aura`.

Сидер должен быть идемпотентным (`updateOrCreate`), чтобы повторный запуск не создавал дублей.

## 6. Логика расчета мотивации (строго совместимо с текущим KPI)

### 6.1 Личная продажа агента (5 и 8)
- Используем тот же кредит продаж, что в KPI (`sales_count` с дробями).
- Порог считается по накопительному `sales_count` агента.

### 6.2 Продажа компании (100)
- Считается суммарный `sales_count` по всем агентам компании Aura.
- Для консистентности с KPI использовать ту же формулу атрибуции продаж.

### 6.3 Период учета
- Milestone мотивации: накопительно, без месячного обнуления.
- KPI-планы периода: неделя/месяц/год.

## 7. Планы периода (week/month/year)

Поддержать установку плана с типом периода:
- `week`
- `month`
- `year`

Минимальные поля в API/DTO плана:
- `period_type` (`week|month|year`)
- `period_start`
- `period_end`
- `metric_key`
- `target_value`
- `scope` (user/branch/company по текущим правам)

### 7.1 План на май
Для текущего этапа зафиксировать в документации, что основной рабочий период запуска планов — `май 2026`:
- месячные планы: `2026-05-01` -> `2026-05-31`.
- недельные планы внутри мая: по календарным неделям.
- годовой план: `2026-01-01` -> `2026-12-31`.

## 8. API (добавить новый модуль, не ломая текущий)

### 8.1 Rules
- `GET /api/motivation/rules`
- `POST /api/motivation/rules`
- `PATCH /api/motivation/rules/{id}`

Права на настройку правил мотивации:
- `rop`, `branch_director`, `admin`, `superadmin`, `owner` (РОП+) — могут создавать и менять правила.
- `agent`, `intern`, `mop`, `manager`, `operator` — только чтение.
- При нарушении: `403 KPI_FORBIDDEN_ROLE_ACTION`.

### 8.2 Achievements
- `GET /api/motivation/achievements?user_id=&status=`
- `POST /api/motivation/recalculate` (ручной/админ пересчет достижений)

### 8.3 Reward issue workflow
- `POST /api/motivation/reward-issues/{achievementId}/assign`
- `PATCH /api/motivation/reward-issues/{id}`

### 8.4 KPI plans by period
- `GET /api/kpi/plans?period_type=week|month|year&date_from=&date_to=`
- `PUT/PATCH /api/kpi/plans/...` с указанием периода.

## 9. RBAC (на базе текущих ролей)
- `agent`: видит только свои достижения/прогресс.
- `mop`: видит свою группу.
- `rop`, `branch_director`: видят свой scope и настраивают мотивации (РОП+).
- `admin/superadmin/owner`: полный доступ и настройка мотиваций (РОП+).

Отдельное правило: управление мотивационными правилами и ручной пересчет — только для РОП+.

## 10. Нотификации
Переиспользовать существующий notifications-модуль:
- категория: `motivation` (уже есть),
- отправка события при достижении 5/8/100,
- дедупликация по `achievement_id`.

## 11. Аудит и консистентность
- Любой пересчет достижений логировать (кто, когда, диапазон дат).
- Если историческая продажа изменена (sold_at, статус, участники), запускать пересчет достижений в затронутом диапазоне.
- Статус `cancelled` использовать при откате достижения из-за исправления данных.

## 12. Acceptance criteria
1. Сидер создает/обновляет 3 правила мотивации (5, 8, 100 company).
2. При достижении агентом суммарного `sales_count >= 5` создается одно достижение "Ташкент".
3. При достижении `sales_count >= 8` создается одно достижение "Умра".
4. При достижении компанией суммарного `sales_count >= 100` создается одно company-достижение "Вечеринка Aura".
5. Повторный пересчет и повторный запуск сидера не создают дублей.
6. Планы поддерживают установку на `week`, `month`, `year`; для запуска задокументирован май 2026.

## 13. Статус на сейчас
Реализацию не начинаем. На текущем шаге зафиксирована только документация и правила для следующего этапа разработки.

## 14. Дополнительные обязательные требования (8 пунктов)

### 14.1 Период плана как обязательная сущность
Для любого мотивационного/планового правила обязательно хранить:
- `period_type`: `week | month | year`
- `date_from`: дата начала периода
- `date_to`: дата окончания периода

Оба поля (`date_from`, `date_to`) обязательны и задают, на какую именно неделю/месяц/год установлен план.

### 14.2 Валидация диапазонов по типу периода
Обязательные проверки:
- `week`: диапазон ровно 7 календарных дней;
- `month`: `date_from` и `date_to` должны лежать в рамках одного календарного месяца;
- `year`: `date_from` и `date_to` должны лежать в рамках одного календарного года.

### 14.3 Выигранные награды хранятся в отдельной таблице
Выигранные (достигнутые) мотивации обязательно сохраняются в отдельной таблице (например `motivation_achievements` / `motivation_wins`), а не только вычисляются на лету.

Минимальные поля:
- `rule_id`
- `user_id` (nullable для company scope)
- `won_at`
- `period_type`
- `date_from`
- `date_to`
- `snapshot_value`
- `status`

### 14.4 Снапшот факта на момент выигрыша
При фиксации выигрыша сохранять снимок факта (`snapshot_value`) и диапазон (`date_from/date_to`) на момент достижения порога.

Это нужно, чтобы история выигрышей оставалась стабильной даже при последующих правках исходных сделок.

### 14.5 Антидубли и идемпотентность
Повторный запуск пересчета и сидеров не должен создавать дубли выигрышей.

Требование к данным:
- уникальные ограничения на комбинации, исключающие повторную фиксацию одной и той же награды в одном и том же scope/периоде.

### 14.6 Статусы жизненного цикла выигрыша
Для записей выигрышей ввести статусы:
- `new`
- `approved`
- `issued`
- `cancelled`

Дополнительно хранить, кто и когда подтвердил/выдал награду.

### 14.7 Правила пересечения планов
Нужно явно зафиксировать политику пересечений:
- запрет пересекающихся планов одного типа периода для одного и того же scope,
- либо строгий приоритет (например, более поздний план перекрывает ранний).

Рекомендуется режим по умолчанию: запрет пересечений и ошибка валидации.

### 14.8 Аудит и таймзона
1. Аудит обязателен:
- кто создал/изменил план,
- кто подтвердил/отменил/выдал выигрыш,
- причина правки/отмены.

2. Единая таймзона расчета обязателена:
- `Asia/Dushanbe` для всех периодов (`week/month/year`) и фиксации `won_at`.
