# Техническое задание: атрибуция продаж в KPI (до 3 продавцов на объект)

## 1. Цель
В рамках KPI корректно фиксировать участие агентов в продаже объекта и считать KPI-метрику `sales_count` с дробным вкладом при совместной продаже.

## 2. Бизнес-правило атрибуции продажи
1. У одного объекта может быть от 1 до 3 продавцов (агентов).
2. Если продажу закрыли несколько агентов, вклад в KPI делится поровну между всеми участвующими агентами.
3. Формула вклада агента по объекту:
   - `agent_sale_credit = 1 / participants_count`
   - где `participants_count` ∈ {1,2,3}.
4. Пример:
   - 1 агент → каждому `1.00`
   - 2 агента → каждому `0.50`
   - 3 агента → каждому `0.33` (в расчетах хранить точность выше, округление только на выдаче)

## 3. Что считать фактом продажи для KPI
Продажа засчитывается строго по:
- дате `properties.sold_at`;
- статусу `properties.moderation_status` в `sold`, `sold_by_owner`, `rented`.

Дата KPI-дня определяется по `sold_at` в таймзоне `Asia/Dushanbe`.

## 4. Источник участников продажи
Основной источник: таблица связей `property_agent_sales` (связь объект ↔ агент-продавец).

Правила:
1. Для KPI использовать только агентов из `property_agent_sales`.
2. Если продажа есть, но участники не заполнены, объект попадает в контроль качества данных (ошибка качества), в KPI не распределяется до исправления.
3. Если участников больше 3 — операция сохранения должна отклоняться (`422`).

### 4.1 Приоритет источников продажи (fixed)
1. Приоритет №1: `property_agent_sales` (явно указанные продавцы объекта).
2. Fallback №2: `properties.agent_id` только если в `property_agent_sales` нет ни одной записи по объекту.
3. Если нет ни `property_agent_sales`, ни `agent_id`, продажа помечается как quality issue и исключается из KPI до исправления.
4. При появлении валидных записей в `property_agent_sales` fallback по `agent_id` больше не применяется.

## 5. Ограничения на редактирование участников продажи

### 5.1 Лимит
- Максимум 3 уникальных агента на один объект.
- Дубли одного и того же `agent_id` запрещены.

### 5.2 Кто может редактировать участников
- Агент/стажер: по действующим правам на редактирование объекта.
- MOP: по scope своей группы.
- ROP/директор филиала/admin/superadmin: по существующему scope и старшинству.

### 5.3 Периодическая консистентность
- Изменение участников продажи у уже проданного объекта должно инициировать пересчет KPI за дату `sold_at`.
- Пересчет должен быть идемпотентным.

## 6. Модель данных (целевое поведение)
Используем существующую таблицу `property_agent_sales` как источник атрибуции.

Требования к данным:
- Уникальность пары (`property_id`, `agent_id`).
- Ограничение количества участников на `property_id` ≤ 3.
- При удалении/изменении состава участников сохранять аудит (кто, когда, до/после).

## 7. Расчет KPI `sales_count`

### 7.1 Дневной расчет
Для каждого агента за дату D:
- выбрать проданные объекты по `sold_at` = D (локальная дата `Asia/Dushanbe`) и валидному `moderation_status`;
- для каждого объекта определить `participants_count` из `property_agent_sales`;
- начислить агенту `1 / participants_count` при его участии;
- итог дня: сумма всех долей.

### 7.2 Формат хранения и точность
- Хранить `sales_count` как decimal (рекомендуется `decimal(8,4)` или выше).
- В API можно отдавать округление до 2 знаков, но расчет и хранение делать с большей точностью.
- Для периодов округление выполнять только после суммирования всех долей.

### 7.3 Отображение
- В daily/weekly/monthly KPI показывать:
  - `sales_count_raw` (точное значение),
  - `sales_count_display` (округленное для UI).

## 8. API-изменения

### 8.1 Property API (где фиксируется продажа)
При переводе объекта в статус продажи поддержать поле участников:
- `sale_agents: [{ agent_id: number }]`

Валидация:
- минимум 1, максимум 3;
- только уникальные `agent_id`;
- агент должен иметь доступ к филиалу объекта (по действующим правилам).

### 8.2 KPI API
- В ответах KPI возвращать дробные продажи.
- В debug/детализации давать разложение по объектам:
  - `property_id`, `sold_at`, `participants_count`, `agent_credit`.

## 9. Контроль качества данных
Добавить проверки качества:
1. Проданный объект без участников продажи.
2. Объект с участниками > 3.
3. Объект с дублями `agent_id` в участниках.

Реакция:
- лог в quality issues;
- исключение некорректного объекта из KPI до исправления;
- уведомление ответственного руководителя.

## 10. RBAC и безопасность
- Все действия по чтению/изменению атрибуции продаж проверяются через существующий branch/group scope.
- Ошибки доступа: `403` с единым форматом ошибок KPI/CRM.

## 11. Критерии приемки
1. Нельзя сохранить более 3 продавцов на объект (`422`).
2. При 3 продавцах каждый получает `0.3333...` в расчете и `0.33` в отображении.
3. Сумма вкладов по одному проданному объекту всегда равна `1.0`.
4. При изменении состава продавцов выполняется корректный пересчет KPI за дату `sold_at`.
5. В weekly/monthly отчетах дробные продажи корректно агрегируются.
6. Проданный объект без продавцов не ломает расчет: фиксируется как quality issue и не учитывается до исправления.

## 12. Тесты (обязательные)
- Unit:
  - деление 1/2/3 и округление;
  - сумма долей = 1.
- Feature:
  - создание/обновление проданного объекта с 1/2/3 агентами;
  - запрет на 4-го агента;
  - пересчет KPI после изменения участников;
  - RBAC-проверки по ролям и scope.
- Regression:
  - существующие отчеты по продажам не ломаются;
  - KPI корректно работает с lock/SLA правилами.

## 13. План внедрения
1. Зафиксировать контракт `sale_agents` в Property API.
2. Добавить/усилить валидацию (<=3, unique, scope).
3. Реализовать сервис атрибуции продаж для KPI.
4. Подключить перерасчет дневного KPI по `sold_at`.
5. Добавить quality checks и аудит.
6. Закрыть unit/feature/regression тестами.

## 14. Миграция исторических данных (Backfill)
Цель: с первого дня релиза получить консистентные KPI по историческим продажам.

Шаги:
1. Выбрать период backfill (дата начала → дата релиза).
2. Взять проданные объекты по `sold_at` + валидные `moderation_status`.
3. Для каждого объекта определить продавцов по приоритету источников (раздел 4.1).
4. Рассчитать доли `1 / participants_count` и пересчитать `sales_count` в daily KPI за соответствующие даты.
5. Сохранить отчет backfill (сколько объектов обработано, сколько quality issues найдено, сколько записей KPI пересчитано).
6. Повторный запуск backfill должен быть идемпотентным.
