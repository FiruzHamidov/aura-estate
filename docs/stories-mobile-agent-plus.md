# Agent+ Stories for Mobile (Android/iOS)

## 1. Назначение
Документ фиксирует продуктовый и технический контракт модуля Stories для мобильных приложений Agent+ (Android/iOS): кто может публиковать, как сторисы отображаются, какие API обязательны для интеграции и какие ограничения должны соблюдаться.

## 2. Scope релиза
В рамках текущей версии:
- Публикация сторис пользователями Agent+.
- Показ публичной ленты сторис в мобильных клиентах.
- Трекинг просмотров.
- Управление собственными сторис и действиями статуса.
- Управление сторис через сайт (web-кабинет/админ-панель) через API-роуты.

Вне текущего scope:
- Контент-модерация второго уровня (жалобы, SLA, очереди проверки).
- Рекомендательный ранжирующий движок.
- Оффлайн-пакетная синхронизация сторис.

## 3. Роли и доступ
Публикация:
- Разрешена только авторизованному пользователю со статусом `active`.
- Запрещена роли `client`.

Просмотр:
- Лента и карточка сторис доступны публично.
- Для неавторизованных просмотров используется `guest_token`.

Управление:
- Владелец может обновлять и удалять свою сторис.
- Доступны статус-действия `archive | hide | republish` по endpoint управления статусом.

## 4. Контентная модель
Типы сторис:
- `media`: сторис с медиа-элементами.
- `property`: сторис из объявления.
- `reel`: сторис из рилса.

Ограничения:
- До `10` элементов `items` в одной сторис.
- Для `reel`: не более `10` одновременно активных reel-сторис на одного автора.
- TTL активной сторис: `24 часа` с момента активации.

## 5. Lifecycle и статусы
Рабочие статусы:
- `draft` -> `active` -> (`archived` | `hidden` | `deleted`)

Правила жизненного цикла:
- При активации вычисляется `expires_at`.
- По истечении `expires_at` сторис исключается из публичной выдачи.
- Удаление выполняется как soft delete (через статус).

## 6. API-контракт для Mobile

### 6.1 Public read APIs
- `GET /api/stories/feed`
  - Назначение: получить публичную ленту сторис, сгруппированную по авторам.
  - Использование: основной источник данных для сторис-трея в мобильке.

- `GET /api/stories/{story}`
  - Назначение: получить полную модель конкретной сторис.
  - Использование: открытие сторис из ленты или deep-link.

- `POST /api/stories/{story}/view`
  - Назначение: зафиксировать просмотр.
  - Для гостя обязателен `X-Guest-Token` (допустимо `guest_token` в body).

### 6.2 Authenticated write APIs
- `POST /api/stories`
  - Назначение: создать сторис типа `media | property | reel`.

- `POST /api/stories/from-property/{property}`
  - Назначение: быстро создать сторис из объявления.

- `POST /api/stories/from-reel/{reel}`
  - Назначение: быстро создать сторис из рилса.
  - Ограничение: контролируется лимит 10 активных reel-сторис.

### 6.3 My stories / maintenance
- `GET /api/my/stories`
  - Назначение: получить собственные сторис (для профиля/кабинета автора).

- `PATCH /api/stories/{story}`
  - Назначение: обновить редактируемые поля сторис.

- `DELETE /api/stories/{story}`
  - Назначение: удалить сторис (soft delete).

- `PATCH /api/stories/{story}/status`
  - Назначение: смена статуса действием `archive | hide | republish`.

### 6.4 Web management routes (обязательно для сайта)
Управление сторис через сайт должно использовать следующие роуты:
- `GET /api/my/stories` — список моих сторис в web-кабинете.
- `POST /api/stories` — создание сторис из web-формы.
- `POST /api/stories/from-property/{property}` — быстрое создание из объявления.
- `POST /api/stories/from-reel/{reel}` — быстрое создание из рилса.
- `PATCH /api/stories/{story}` — редактирование сторис.
- `DELETE /api/stories/{story}` — удаление сторис.
- `PATCH /api/stories/{story}/status` — архивировать/скрыть/перепубликовать.
- `GET /api/admin/stories` — список сторис для админ-интерфейса.
- `PATCH /api/admin/stories/{story}/status` — админское изменение статуса.

## 7. Нефункциональные требования
Rate limiting:
- Создание сторис: `30` запросов в час на пользователя.
- Трекинг просмотров: `120` запросов в минуту.

Производительность клиента:
- Лента должна рендериться на базе `feed` без дополнительных обязательных запросов.
- Детальная подгрузка по `GET /api/stories/{story}` выполняется лениво по событию открытия.

## 8. Ошибки и обработка на мобильной стороне
Рекомендуемая обработка:
- `401` — неавторизован для write-операций, инициировать re-auth.
- `403` — недостаточно прав (например, роль `client`), показать бизнес-сообщение.
- `404` — сторис недоступна/истекла/не найдена, удалить из локального UI-кеша.
- `422` — валидационные/бизнес-ограничения (включая лимит reel-сторис), показать текст ошибки пользователю.
- `429` — превышен rate limit, применить retry с backoff.

## 9. Референс интеграционного потока
1. Автор создает сторис через `POST /api/stories` или fast-create endpoint.
2. Бэкенд возвращает сущность сторис с полями статуса и срока жизни (`expires_at`).
3. Мобильный клиент получает ленту через `GET /api/stories/feed`.
4. При фактическом просмотре отправляется `POST /api/stories/{story}/view`.
5. После истечения TTL сторис исчезает из публичной выдачи.

## 10. Критерии приемки
- Пользователь Agent+ со статусом `active` (кроме `client`) может создать сторис.
- Android/iOS корректно отображают публичную ленту через `GET /api/stories/feed`.
- Просмотры фиксируются через endpoint `view`.
- Лимит `10` активных reel-сторис на автора соблюдается.
- Истекшие сторис не возвращаются в публичной ленте.
