# Техническое задание: доработка KPI-модуля (индивидуальные планы и ежедневная отчетность)

## 1. Цель
Доработать KPI-модуль CRM так, чтобы KPI-план настраивался индивидуально для каждого сотрудника, ежедневная отчетность заполнялась по 5 целевым показателям, а руководители могли контролировать выполнение в своих зонах ответственности.

## 2. Бизнес-контекст
Система должна поддерживать:
- персональные KPI-планы для каждого сотрудника;
- ежедневные отчеты по KPI;
- автоматический сбор части метрик из CRM;
- ручной ввод метрик, которые не считаются автоматически;
- контроль и аналитику в разрезе оргструктуры.

## 3. Роли и область видимости

### 3.1 Сотрудник (агент/стажер)
- Видит и редактирует только свои ежедневные KPI-отчеты (до закрытия периода, если период залочен).
- Видит свой KPI-план и процент выполнения.

### 3.2 MOP
- Видит ежедневные и итоговые KPI-отчеты сотрудников своей группы.
- Не имеет доступа к сотрудникам других групп.
- Может редактировать отчеты своей группы только в незалоченном периоде.

### 3.3 ROP
- Настраивает KPI-планы для сотрудников всего филиала.
- Видит ежедневные и итоговые KPI-отчеты по всему филиалу.
- Не имеет доступа к чужим филиалам.
- Может редактировать отчеты своего филиала даже в залоченном периоде.

### 3.4 Директор филиала
- Настраивает KPI-планы для сотрудников своего филиала.
- Видит ежедневные и итоговые KPI-отчеты по своему филиалу.
- Не имеет доступа к чужим филиалам.
- Может редактировать отчеты своего филиала даже в залоченном периоде.

### 3.5 Admin/Superadmin (владелец)
- Настраивает KPI-планы для любых сотрудников.
- Глобальный доступ ко всем филиалам и группам.
- Может редактировать отчеты в любых периодах, включая залоченные.

## 4. KPI-метрики (обязательные)
Для каждого сотрудника должны учитываться 5 метрик:
1. Объекты: количество добавленных объектов.
2. Показы: количество добавленных показов.
3. Реклама: количество размещений рекламы на внешних площадках.
4. Звонки: количество совершенных звонков.
5. Продажи: количество проданных объектов.

## 5. Источники данных по метрикам

### 5.1 Автоматически из CRM
- `objects_count` (Объекты): считать по `properties`, созданным сотрудником за дату отчета.
- `shows_count` (Показы): считать по `bookings`, созданным сотрудником за дату отчета.
- `sales_count` (Продажи): считать строго по полю `properties.sold_at` (дата продажи) в диапазоне дня отчета с привязкой к ответственному сотруднику. В расчет включать только записи с `moderation_status` из набора `sold`, `sold_by_owner`, `rented`.

### 5.2 Вручную
- `ads_count` (Реклама): вводится пользователем вручную.
- `calls_count` (Звонки): вводится пользователем вручную.

## 6. Индивидуальный KPI-план

### 6.1 Требования
- План задается на сотрудника (а не только на роль).
- План хранит целевые дневные значения по всем 5 метрикам.
- План может иметь период действия (`effective_from`/`effective_to`), чтобы сохранялась история изменений.
- При отсутствии персонального плана используется fallback на ролевой/дефолтный план (если такой существует).

### 6.2 Поля плана (минимум)
- `user_id`
- `branch_id`
- `branch_group_id` (nullable)
- `objects_plan`
- `shows_plan`
- `ads_plan`
- `calls_plan`
- `sales_plan`
- `effective_from`
- `effective_to` (nullable)
- `created_by`, `updated_by`
- timestamps

## 7. Ежедневная отчетность

### 7.1 Создание/обновление отчета
- На каждый день и пользователя создается одна запись daily KPI report.
- Автоматические метрики подтягиваются системой.
- Ручные метрики (`ads_count`, `calls_count`) заполняются пользователем.
- Разрешить повторное редактирование по правилам lock-периода:
  - Агент/стажер и MOP: редактирование запрещено в залоченном периоде.
  - ROP, директор филиала, admin, superadmin: редактирование разрешено даже в залоченном периоде.

### 7.2 Поля дневного отчета (минимум)
- `report_date`
- `user_id`
- `branch_id`
- `branch_group_id`
- Факт:
  - `objects_fact_auto`
  - `shows_fact_auto`
  - `sales_fact_auto`
  - `ads_fact_manual`
  - `calls_fact_manual`
- План на дату (snapshot из актуального плана):
  - `objects_plan`
  - `shows_plan`
  - `ads_plan`
  - `calls_plan`
  - `sales_plan`
- Выполнение (можно хранить или считать на лету):
  - `objects_percent`
  - `shows_percent`
  - `ads_percent`
  - `calls_percent`
  - `sales_percent`
  - `total_percent`
- `comment` (nullable)
- `submitted_at`
- `is_finalized` (bool, default `false`) — признак зафиксированного (снимка) итога дня.
- `finalized_at` (nullable datetime)
- `finalized_by` (nullable user_id)

## 8. Расчет выполнения KPI

### 8.1 По метрике
Для каждой метрики:
- `metric_percent = min(fact / plan, 1.0) * 100`, если `plan > 0`.
- Если `plan = 0`:
  - при `fact = 0` считать 100%;
  - при `fact > 0` считать 100% (или отдельно согласовать правило; по умолчанию 100%).

### 8.2 Общий KPI за день
- По умолчанию: среднее арифметическое 5 метрик.
- Опционально: поддержать веса метрик (конфигурируемо).

### 8.3 Итог за период
- Дневные значения агрегируются в weekly/monthly/custom периодах.
- Для периода считать:
  - сумму факта по каждой метрике,
  - сумму плана по каждой метрике,
  - процент выполнения по каждой метрике,
  - общий процент выполнения за период.
- Округление итоговых процентов и итоговых значений выполнять только после суммирования по периоду.

### 8.4 Политика округления
- Для расчетов и хранения KPI использовать decimal с точностью не ниже `decimal(8,4)`.
- В UI отображать значения с округлением до 2 знаков после запятой.
- В API при необходимости отдавать оба формата:
  - `*_raw` — точное значение для машинной обработки;
  - `*_display` — округленное значение для интерфейса.

## 9. Правила доступа (RBAC)
Все операции чтения/записи должны проверяться по branch/group scope:
- Сотрудник: только self.
- MOP: только свой `branch_group_id`.
- ROP: только свой `branch_id`.
- Директор филиала: только свой `branch_id`.
- Admin/Superadmin: без ограничений по филиалу/группе.
- При отсутствии прав — единый `403 Forbidden`.

## 10. API (целевое поведение)

### 10.1 Планы
- `GET /api/kpi/plans?user_id=...&date=...` — получить эффективный план сотрудника на дату.
- `PUT/PATCH /api/kpi/plans/{user_id}` — установить/обновить персональный план.

### 10.2 Дневные отчеты
- `GET /api/kpi/daily?date=...&user_id=...` — получить отчет(ы) за день по scope.
- `POST /api/kpi/daily` — создать/сохранить дневной отчет (ручные поля + автопересчет авто-полей).

### 10.3 Сводка
- `GET /api/kpi/summary?date_from=...&date_to=...&branch_id=...&branch_group_id=...&user_id=...`
- Возвращает выполнение KPI за период в разрезах:
  - сотрудник,
  - группа,
  - филиал.

## 11. UI/UX требования (минимум)
- Форма дневного отчета:
  - авто-поля только для чтения (Объекты, Показы, Продажи);
  - ручные поля для ввода (Реклама, Звонки);
  - отображение плана, факта и процента выполнения.
- Экран руководителя:
  - таблица сотрудников по scope;
  - фильтры по периоду, группе, сотруднику;
  - статус выполнения (цвета/состояния).

## 12. Аудит и прозрачность
- Логировать изменения KPI-планов (кто, когда, что изменил).
- Логировать ручные правки в дневных отчетах (кто и когда изменил `ads/calls`).
- Логировать операции `finalize/reopen` KPI-дня (кто, когда, причина).

## 13. Нефункциональные требования
- Таймзона расчетов: `Asia/Dushanbe`.
- Идемпотентность сохранения дневного отчета: повторный POST/PUT за ту же дату и user обновляет существующую запись.
- Производительность: сводка за месяц по филиалу должна строиться без N+1 и с пагинацией/батчингом при необходимости.

## 14. SLA по времени сабмита
- Дедлайн сабмита дневного отчета: `23:59` по таймзоне `Asia/Dushanbe` для даты отчета.
- До дедлайна сотрудник (агент/стажер) может создавать и редактировать свой отчет.
- После дедлайна:
  - сотрудник (агент/стажер) и MOP не могут редактировать отчет за этот день;
  - ROP, директор филиала, admin и superadmin могут редактировать отчет (в рамках своего scope для ROP/директора).
- При попытке редактирования после дедлайна неразрешенной ролью API возвращает `403` с кодом бизнес-ошибки (см. раздел API-контрактов ошибок).
- Для воспроизводимости отчетов поддержать `finalized`-снапшот KPI-дня:
  - после завершения дня (и/или по регламенту) отчет помечается `is_finalized=true`;
  - KPI-сводки по закрытым периодам строятся из finalized-данных;
  - изменение finalized-данных допускается только для ROP/директора/admin/superadmin с обязательным аудитом и причиной.

## 15. API-контракты ошибок (единый формат)
Для всех KPI-эндпоинтов использовать единый JSON-формат ошибок:

```json
{
  "code": "ERROR_CODE",
  "message": "Human readable message",
  "details": {},
  "trace_id": "uuid-or-request-id"
}
```

Правила по статусам:
- `403 Forbidden` — нет прав/нарушен scope/запрещено политикой lock или SLA.
- `422 Unprocessable Entity` — ошибка валидации входных данных.
- `409 Conflict` — конфликт состояния (например, пересечение периодов KPI-плана, конфликт блокировки периода, duplicate logical constraint).

Минимальные коды ошибок:
- `KPI_FORBIDDEN_SCOPE` — доступ вне branch/group scope.
- `KPI_FORBIDDEN_LOCKED_PERIOD` — редактирование запрещено lock-политикой роли.
- `KPI_FORBIDDEN_DEADLINE_PASSED` — дедлайн сабмита истек для роли.
- `KPI_VALIDATION_FAILED` — ошибка валидации полей.
- `KPI_PLAN_PERIOD_CONFLICT` — пересечение/конфликт периода KPI-плана.
- `KPI_CONFLICT` — прочий бизнес-конфликт.
- `KPI_FINALIZED_EDIT_FORBIDDEN` — попытка редактирования finalized-данных без достаточной роли.

## 16. Критерии приемки
1. MOP не может назначать персональные KPI-планы (получает `403`).
2. ROP может назначить персональный план сотруднику своего филиала и не может назначить для чужого филиала.
3. Директор филиала может назначить персональный план сотруднику своего филиала и не может назначить для чужого филиала.
4. Admin/Superadmin может назначить персональный план любому сотруднику.
5. Сотрудник создает дневной отчет: авто-метрики подтягиваются из CRM, ручные метрики сохраняются.
6. Дневной KPI-процент считается корректно по всем 5 метрикам.
7. Итоговый KPI за неделю/месяц считается корректно по суммарному плану/факту.
8. Все нарушения scope возвращают `403`.
9. В отчетах MOP видит только свою группу, ROP/директор филиала — только свой филиал.

## 17. Этапы реализации
1. Модель данных: персональные планы + снапшот плана в daily report.
2. Сервис расчета авто-метрик (properties/bookings/sold).
3. API для управления планами.
4. API для дневных отчетов (save/read).
5. API сводной аналитики KPI.
6. RBAC/scope проверки и аудит.
7. Реализация finalized-снапшота KPI-дня и правил его редактирования.
8. Тесты (feature + unit) по ролям, расчетам, округлению и источникам данных.

## 18. Риски и уточнения
- Правило расчета `sales_count` зафиксировано: источник даты продажи — поле `properties.sold_at`.
- Нужно подтвердить, считать ли KPI как среднее или как взвешенную сумму (если веса обязательны).
- Правило lock-периода зафиксировано: ограничение действует только для агента/стажера и MOP; ROP и вышестоящие роли могут редактировать в залоченном периоде.
- Для исторической консистентности требуется backfill KPI-данных по продажам за прошлые периоды (см. отдельное ТЗ по атрибуции продаж).
