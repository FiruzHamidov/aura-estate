# ТЗ: Автоматический отчет для РОП по МОП и их агентам

## 1. Цель
Сделать автоматическую систему ежедневной/недельной отчетности, чтобы РОП получал готовый отчет по МОП и их агентам без ручного копирования данных из Telegram/Excel.

## 3. Текущая логика KPI (как в Excel)
Показатели по строке сотрудника за день:
- `Кабул`
- `Показ`
- `Реклама`
- `Залог`
- `Звонок`
- `Привод`
- `Сделка`

План и вес (лист `Планы KPI`):
- Реклама: план `20`, вес `0.15`
- Звонок: план `30`, вес `0.20`
- Кабул: план `5`, вес `0.15`
- Показ: план `2`, вес `0.20`
- Привод: план `1`, вес `0.10`
- Залог: план `1`, вес `0.10`
- Сделка: план `1`, вес `0.10`

Формула KPI за день:
`KPI_day = SUM((Факт_i / План_i) * Вес_i)`

Статус за день:
- `KPI >= 1.0` -> `Выполнено`
- `0.8 <= KPI < 1.0` -> `Контроль`
- `0.6 <= KPI < 0.8` -> `Слабый`
- `KPI < 0.6` -> `Срочно`

Неделя:
- агрегирование по `Неделя + Сотрудник`
- суммы по показателям и `KPI_week = AVG(KPI_day)`
- статус недели:
  - `KPI_week >= 1.0` -> `Неделя выполнена`
  - `0.8 <= KPI_week < 1.0` -> `Под контролем`
  - `0.6 <= KPI_week < 0.8` -> `Риск`
  - `KPI_week < 0.6` -> `Провал`

## 3.1 Бизнес-правила KPI (уточнение)
- Дневной KPI и его веса задает РОП (через настройки `Планы KPI`/админ-панель).
- Недельный и месячный KPI рассчитываются автоматически на основе дневного KPI.
- В неделе принимается `6` рабочих дней как плановая норма.
- Если сотрудник работал в воскресенье, данные воскресенья учитываются в факте, но статус выполнения недели определяется по достижению плановых показателей за `6` рабочих дней.
- Если план выполнен раньше конца недели/месяца, период получает статус `Успешно` (досрочно выполнен).
- Логика итогов:
  - `KPI_week` = расчет из дневных KPI за период недели с учетом нормы 6 рабочих дней.
  - `KPI_month` = расчет из дневных KPI за календарный месяц.

## 5. Что обязательно тянуть автоматически
Ниже перечислены KPI-поля, которые система должна собирать и хранить.

1. `Звонок`
Источник: телефония/коллтрекинг/CRM звонков.
Правило: количество валидных исходящих/целевых звонков за день по сотруднику.

2. `Реклама`
Источник: ручной ввод МОП (внешние площадки).
Правило: МОП вручную указывает число опубликованных объявлений за день.

3. `Кабул` (принятые лиды)
Источник: CRM лидов.
Правило: число принятых/обработанных лидов за день.

4. `Показ`
Источник: CRM активности/события.
Правило: количество подтвержденных показов объектов за день.

5. `Привод` (встреча в офисе)
Источник: ручной ввод МОП.
Правило: МОП вручную указывает количество клиентов, принятых в офисе (проведенные встречи) за день.

6. `Залог`
Источник: CRM сделок/договоров.
Правило: количество зарегистрированных залогов за день.

7. `Сделка`
Источник: CRM сделок.
Правило: количество завершенных сделок за день.

## 5.1 Справочник типов задач (обязательный)
Для корректного расчета KPI в CRM вводится справочник `task_types`.
Подсчет каждой KPI-метрики должен идти по задачам соответствующего типа.

Минимальный справочник типов:
- `CALL` (`Звонок`) — использовать для подсчета показателя `Звонок`.
- `LEAD_ACCEPT` (`Кабул`) — использовать для подсчета показателя `Кабул`.
- `MEETING_OFFICE` (`Встреча в офисе`) — использовать для подсчета показателя `Привод/Встреча`.
- `SHOWING` (`Показ`) — использовать для подсчета показателя `Показ`.
- `DEPOSIT` (`Залог`) — использовать для подсчета показателя `Залог`.
- `DEAL_CLOSED` (`Сделка`) — использовать для подсчета показателя `Сделка`.
- `AD_PUBLICATION` (`Публикация рекламы`) — тип для фиксации рекламной активности (на текущем этапе значение может заполняться вручную МОП).

Правило для звонков:
- Чтобы звонок попал в KPI, в системе должна быть создана задача типа `CALL` (`Звонок`) и она должна иметь валидный статус выполнения.

## 5.2 Задачи в CRM (расширенное ТЗ)
Типы задач должны покрывать не только KPI, но и операционную работу CRM, показы и объявления.

Группы типов задач:
- KPI-задачи: `CALL`, `LEAD_ACCEPT`, `MEETING_OFFICE`, `SHOWING`, `DEPOSIT`, `DEAL_CLOSED`, `AD_PUBLICATION`.
- CRM-процесс: `FOLLOW_UP`, `DOCUMENT_REQUEST`, `CONTRACT_PREP`, `CLIENT_QUALIFICATION`, `REASSIGNMENT`.
- Объявления: `AD_CREATE`, `AD_EDIT`, `AD_PUBLISH`, `AD_REFRESH`, `AD_ARCHIVE`.
- Показ/объект: `SHOWING_PLAN`, `SHOWING_CONFIRM`, `SHOWING_COMPLETE`, `SHOWING_CANCEL`, `POST_SHOWING_FEEDBACK`.

Обязательные поля задачи:
- `task_id`
- `task_type_code` (из справочника)
- `title`
- `description`
- `assignee_id`
- `creator_id`
- `related_entity_type` (`lead|client|deal|property|ad|showing`)
- `related_entity_id`
- `due_at`
- `completed_at`
- `status` (`new|in_progress|done|canceled|overdue`)
- `result_code` (справочник результата выполнения)
- `source` (`manual|system|integration`)

Обязательные связи:
- задача по показу должна ссылаться на сущность `showing` и/или `property`.
- задача по рекламе должна ссылаться на сущность `ad`.
- задачи по клиенту/сделке должны ссылаться на `lead|client|deal`.

Правила учета в KPI:
- В KPI попадают только задачи нужного типа с `status=done`.
- Если задача отменена (`canceled`) или просрочена без выполнения, в KPI не учитывается.
- Для аудита хранить историю изменения `status`, `assignee`, `due_at`, `completed_at`.

## 6. Поля ручного ввода (текущий этап)
На текущем этапе вручную заполняются только:
- `Реклама` (заполняет МОП по внешним площадкам)
- `Встреча` / `Привод` (заполняет МОП: сколько приняли в офисе)
- `Комментарий` (заполняет МОП/РОП)

Все остальные KPI-поля и расчеты должны заполняться/считаться автоматически.

## 7. Расчеты и отчеты
Система должна автоматически строить:
- ежедневный отчет по каждому агенту
- свод по МОП (сумма команды + средний KPI)
- недельный итог (аналог листа `Недельный итог`)
- дашборд РОП (аналог листа `Дашборд`)
- рейтинг сотрудников/команд по KPI

## 8. Выгрузка для РОП
Форматы выдачи:
- web-дашборд
- ежедневная отправка в Telegram (08:30 локального времени)
- недельный отчет по понедельникам (09:00)
- выгрузка в Excel по шаблону, совместимому с текущим форматом

Минимум в сообщении РОП:
- топ-3 и низ-3 по KPI
- сотрудники со статусом `Срочно/Провал`
- динамика к вчера/прошлой неделе
- итог по МОП и по отделу

## 9. Контроль качества данных
Проверки перед публикацией отчета:
- нет дублей по `дата + сотрудник + метрика`
- нет отрицательных значений
- все сотрудники привязаны к МОП
- полнота данных по обязательным источникам

При сбое интеграции:
- помечать метрику статусом `source_error`
- отправлять уведомление ответственному
- не блокировать остальные корректные метрики

## 9.1 Заморозка периода (lock)
- После закрытия периода (`day`, `week`, `month`) данные фиксируются и получают статус `locked`.
- Изменение `locked`-периода разрешено только через корректировку.
- Каждая корректировка обязана писать аудит:
  - `period_type`, `period_key`
  - `entity_id` (агент/МОП/группа)
  - `field_name`
  - `old_value`
  - `new_value`
  - `changed_by`
  - `changed_at`
  - `reason`
- Во фронте показывать признак, что период заморожен, и историю корректировок.

## 9.2 Алерты раннего риска
- Если у сотрудника `KPI_day < 0.8` два рабочих дня подряд, система создает алерт `early_risk`.
- Получатели алерта: соответствующий МОП и РОП.
- Каналы: in-app уведомление + Telegram.
- В алерте должны быть:
  - сотрудник
  - последние 2 дня и значения KPI
  - метрики с наибольшим отставанием (`progress_pct` минимальный)
  - ссылка на карточку сотрудника в отчете

## 10. Интеграционные требования
- API-коннекторы к CRM/телефонии/рекламным источникам
- инкрементальная загрузка (по `updated_at`)
- повторная дозагрузка при ошибках (retry)
- хранение истории пересчетов KPI

## 10.1 Требования фронт/бэкенд по периодам
Фронт должен уметь переключать период отображения:
- `День`
- `Неделя`
- `Месяц`

Требование к бэкенду: полностью поддержать эти срезы в API и агрегациях.

Минимальный контракт API:
- фильтры: `period_type=day|week|month`, `date_from`, `date_to`, `mop_id`, `agent_id`, `group_id`
- группировки: по `дате` (день), по `номеру недели` (неделя), по `YYYY-MM` (месяц)
- метрики в ответе: все 7 KPI-показателей + `kpi_value` + `status`
- для каждой метрики и итогового KPI возвращать отдельно:
  - `fact_value` (факт)
  - `target_value` (цель/план)
  - `progress_pct` (процент выполнения = `fact_value / target_value * 100`)
- детализация: `agent`, `mop`, `group`, `department_total`

Правила агрегации:
- `day`: факт за календарный день
- `week`: сумма метрик за неделю + расчет `KPI_week` из дневных KPI, плановая норма недели = `6` рабочих дней
- `month`: сумма метрик за месяц + расчет `KPI_month` из дневных KPI календарного месяца
- если недельный/месячный план выполнен досрочно, статус периода = `Успешно` (досрочно выполнен)

## 11. Права доступа
- РОП: полный просмотр всех МОП/агентов
- Директор филиала: просмотр KPI и отчетности только по своему филиалу
- МОП: только своя команда
- Агент: только свои показатели
- Админ: полный доступ ко всем филиалам, всем KPI и всей отчетности + настройка планов/весов KPI

## 12. Этапы реализации
1. Описать и утвердить mapping полей из каждой системы в 7 KPI-метрик.
2. Сделать слой загрузки `daily_metrics_raw`.
3. Сделать нормализацию и расчет KPI (день/неделя).
4. Собрать дашборд + Telegram-рассылку.
5. Провести параллельный прогон с текущим Excel 2 недели.
6. Перевести РОП на полностью автоматический отчет.

## 13. Критерии приемки
- не менее `95%` метрик заполняются автоматически без ручного вмешательства
- расхождение с контрольным Excel не более `1%` по агрегатам
- ежедневный отчет формируется и отправляется по расписанию
- РОП видит свод по МОП и детализацию до агента в 2 клика

## 14. Что уже можно автоматизировать сразу (по данным файла)
Сразу автоматизируются расчеты и агрегации:
- KPI за день
- статус за день
- номер недели
- недельные суммы по сотруднику
- недельный средний KPI
- недельный статус
- общий дашборд отдела

Ручным сейчас остается только первичный ввод фактов, который должен быть заменен интеграциями из систем-источников.
